# 🔌 프론트 ↔ 백엔드 연결 안내 (To. 정)

> 백엔드 API 연결 방법입니다. 아래대로 요청 보내주시면, 챗봇 답변과 채워진 서식을 돌려드려요.
> 화면/상태관리는 프론트, 질문 처리/서식/PDF는 백엔드가 담당합니다.
> 필드명 상세는 별첨 `FIELD_SPEC.md` 참고.

---

## 0. 서버 주소

```
베이스 URL: https://__________.ngrok.io     ← (백엔드가 서버 띄운 뒤 전달)
```

> 코랩+ngrok이라 발표 때마다 주소가 바뀝니다. 그때그때 공유드릴게요.

## 1. 엔드포인트 3개

| 메서드 | 주소 | 역할 |
| --- | --- | --- |
| POST | `/api/chat` | 대화 (서식 채우기 / 상담) |
| POST | `/api/render` | 채워진 서식 → 한국어 제출용 PDF |
| POST | `/api/ocr-arc` | 신분증 사진 → 정보 추출 |

## 2. ⭐ /api/chat — 가장 많이 쓰는 것

### 프론트가 할 일
1. 사용자(또는 페르소나)가 입력한 메시지를 아래 형식으로 전송
2. 받은 응답을 채팅창 + 서식 화면에 표시
3. `updated_form_state`를 **다음 요청에 그대로 다시 보내기** (상태 누적)

### 📤 보낼 것 (Request)
```json
{
  "scenario": "remittance",
  "user_message": "Tôi muốn gửi 1000 USD cho mẹ tôi",
  "current_form_state": {
    "sender_name_eng": "NGUYEN VAN AN",
    "remittance_amount": null,
    "beneficiary_name": null
  },
  "history": [],
  "language": "Chinese",
  "stage": ""
}
```

- `scenario`: **버튼에 따라** 셋 중 하나
  - 계좌개설 버튼 → `"account_opening"`
  - 외화송금 버튼 → `"remittance"`
  - 일반상담 버튼 → `"troubleshooting"`
- `user_message`: 사용자 입력 (어떤 언어든 OK). **선택지 버튼을 눌렀으면 그 번호**(예: "2")
- `current_form_state`: 지금까지 채워진 서식 상태 (처음엔 `{}` 또는 null들)
- `history`: 이전 대화 (선택, 없으면 `[]`)
- `language`: 선택한 언어 (Korean/English/Chinese/Japanese/Vietnamese/Indonesian). 비면 자동 감지
- `stage`: 진행 단계. **응답에서 받은 stage를 그대로 다시 보냄** (처음엔 `""`)

### 📥 받을 것 (Response)
```json
{
  "bot_reply": "거래 목적을 선택해 주세요:\n1. 급여/생활비  2. 저축/투자 ...",
  "updated_form_state": { "customer_name": "NGUYEN", "birth_date": "2004-07-22" },
  "form_complete": false,
  "stage": "group:purpose",
  "choices": ["급여/생활비", "저축/투자", "보험료", "공과금", "..."],
  "choice_group": "purpose",
  "consent_items": [],
  "saved": {}
}
```

- `bot_reply`: 챗봇 답변 → **채팅창에 표시** (사용자 언어로 나감)
- `updated_form_state`: 갱신된 서식 → **빈칸에 표시** + 다음 요청에 다시 보내기
- `form_complete`: `true`면 완료 → **"PDF 보기" 버튼 활성화**
- `stage`: 현재 단계. **받은 값을 다음 요청에 그대로 다시 보내기** (`""` / `"product"` / `"group:xxx"` / `"consent"` / `"done"`)
- `choices`: ⭐ **번호 선택지 배열**. 있으면 **버튼으로 만들기** (아래 참고)
- `choice_group`: 어떤 선택인지 (product/occupation/carrier/mailing/purpose/source/overseas)
- `consent_items`: 약관 항목 배열. `stage:"consent"`일 때 약관 목록 표시
- `saved`: 완료 시 DB 저장 결과 `{saved, id, summary}`

### ⭐ choices(버튼) 처리 — 중요
계좌개설은 직업·통신사·상품·거래목적 등을 **번호 선택지**로 물어봅니다.
응답에 `choices`가 있으면 버튼으로 만들고, 누른 번호를 `user_message`로 보내세요.

```javascript
const res = await response.json();

if (res.choices && res.choices.length > 0) {
  // 각 선택지를 버튼으로
  res.choices.forEach((label, i) => {
    만들기: 버튼(label), 클릭 시 → user_message = String(i + 1) 로 전송
  });
} else {
  // 일반 텍스트 입력창
}
```

예: `choices: ["급여/생활비", "저축/투자", ...]` 에서 2번째 버튼 누름
→ 다음 요청 `user_message: "2"`, `stage: "group:purpose"`(받은 값 그대로)

### 약관 동의 처리 (stage: "consent")
`stage`가 `"consent"`면 `consent_items`의 약관 목록을 보여주고,
사용자가 동의하면 `user_message: "동의"`(또는 "agree"), `stage: "consent"`로 보내세요.

### 핵심 동작
- 백엔드는 **상태를 기억 안 함**. 매 요청에 `current_form_state` + `stage`를 통째로 보내주세요.
  (받은 `updated_form_state` → 다음 `current_form_state`, 받은 `stage` → 다음 `stage`)
- `troubleshooting`이면 서식이 없으므로 `form_state`는 `{}`, `choices`도 없음.

## 3. /api/render — PDF 생성 (form_complete가 true일 때)

### 📤 보낼 것
```json
{
  "scenario": "account_opening",
  "form_state": { ...채워진 전체 form_state... },
  "signatures": {
    "signature": "<base64 PNG>"
  },
  "checks": {}
}
```
- `signatures`: ⭐ **서명 하나만 `"signature"` 키로 보내면 됩니다.**
  백엔드가 시나리오별 필요한 모든 서명 칸에 자동으로 복사합니다.
  (계좌개설 3곳, 외화송금 2곳 — 프론트는 신경 안 써도 됨)
- `checks`: 추가 체크 (보통 비움 `{}`. 그룹/약관 체크는 form_state에 "V"로 넣음)

> 서명 패드에서 받은 PNG를 base64로 바꿔 `signature` 하나로 보내면 끝.
> (특정 칸만 다르게 넣고 싶으면 아래 개별 키로도 가능하지만, 보통은 `signature` 하나면 됩니다.)

**서명 key 목록 (시나리오별):**

외화송금 (2개):
```
req_applicant_signature       (신청인 서명)
req_withdrawal_signature      (출금계좌 예금주 서명)
```

계좌개설 (필수 3개 + 선택 4개):
```
[필수 - 상품 1개 가입 시]
req_applicant_signature_p0        (신청인 서명, 1페이지 맨아래)
req_confirm_info_inquiry_sig      (①번 상품 줄 확인 서명)
req_depositor_final_signature     (예금주 최종 서명, 2페이지 맨아래)
[선택 - 보통 안 보냄]
req_confirm_group_share_sig       (②번 상품 줄 - 상품 2개째 가입 시만)
req_confirm_product_explain_sig   (③번 상품 줄 - 상품 3개째 가입 시만)
opt_ebanking_signature            (전자금융 신청 시)
opt_card_signature                (IC카드 신청 시)
opt_renewal_applicant_signature   (자동재예치 신청 시)
```
> ⚠️ 상품을 1개만 가입하면 ②③번 줄(상품·서명·체크박스)은 전부 비웁니다.
>   즉 `req_confirm_group_share_sig`, `req_confirm_product_explain_sig`는 보내지 마세요.
> 데모에선 같은 서명 이미지를 모든 서명란에 넣어도 됩니다.
> 최소한 `req_applicant_signature_p0`(계좌개설) / `req_applicant_signature`(송금)만 넣어도 PDF는 생성됩니다.

### 📥 받을 것
- 완성된 **PDF 파일** (`application/pdf`) → 다운로드/미리보기로 표시

> 확인용은 화면에 영어로(프론트), 제출용 PDF는 한국어(이 API). 서로 분리됨.

## 4. /api/ocr-arc — 신분증 인식

### ⭐ 신분증 받는 흐름 (중요)
계좌개설을 시작하면, 백엔드가 **이름·생년월일을 받기 위해 신분증을 먼저 요청**합니다.

```
계좌개설 첫 메시지 전송
   ↓
백엔드 응답: { stage: "id_card", request_id_card: true, bot_reply: "신분증을 올려주세요" }
   ↓
프론트: stage가 "id_card"이거나 request_id_card가 true면
        → 텍스트 입력창 대신 [사진 업로드 창] 표시
   ↓
사용자가 신분증 사진 업로드 → /api/ocr-arc 로 파일 전송 (아래)
   ↓
받은 name/birth를 form_state에 넣고,
   /api/chat 을 다시 호출 (user_message: "업로드 완료" 등, stage: "id_card" 그대로)
   ↓
백엔드가 이름·생년월일 확인 후 → 다음 단계(전화번호)로 진행
```

**핵심:**
- 응답에 `request_id_card: true` (또는 `stage: "id_card"`) → 업로드 창 띄우기
- OCR 결과(name, birth_date)를 `current_form_state`의 `customer_name`, `birth_date`에 넣기
- 그 다음 `/api/chat` 호출 시 `stage: "id_card"` 그대로 보내면 → 백엔드가 다음으로 진행
- 사용자가 신분증을 못 올려도, 한 번 stage="id_card"로 호출하면 백엔드가 그냥 다음으로 넘어감 (멈추지 않음)

### 📤 보낼 것 (multipart/form-data)
신분증 이미지 파일

### 📥 받을 것
```json
{
  "status": "success",
  "data": {
    "name": "NGUYEN THI LINH",
    "birth_date": "2002-03-15",
    "arc_number": "990101-6xxxxxx",
    "visa_type": "D-4",
    "nationality": "Vietnam"
  }
}
```
> 추출된 name, birth_date를 form_state에 넣어 다음 대화에 사용. **로컬 처리** (EasyOCR+Qwen).

**보내는 법 (multipart/form-data):**
```javascript
const fd = new FormData();
fd.append("file", 신분증이미지파일);
fd.append("language", "Vietnamese");  // 선택한 언어 (완료 메시지를 그 언어로 받음)
const r = await fetch(url + "/api/ocr-arc", {
  method: "POST",
  headers: {"ngrok-skip-browser-warning": "true"},
  body: fd   // Content-Type은 자동 설정 (직접 넣지 말 것)
});
const res = await r.json();
// {status, data:{name, birth_date, arc_number, ...}, message: "완료 메시지(해당 언어)"}
// res.message 를 채팅창에 표시하면 됨 (하드코딩 X)

// 결과를 form_state에 반영
formState.customer_name = res.data.name;
formState.birth_date = res.data.birth_date;
// 그 다음 /api/chat 을 stage:"id_card"로 호출하면 다음 단계로 진행
```
★ 신분증은 외부 API로 안 나가고 전부 로컬 처리됩니다 (보안).

## 5. 한 번의 대화 흐름 (예시)

```
1. 사용자가 "외화송금" 버튼 클릭
   → 프론트: scenario="remittance", form_state={} 준비

2. 사용자 입력 "I'm John, send 1000 USD"
   → POST /api/chat
   ← bot_reply="수취인이 누구세요?", form_state={sender:..., amount:1000}, complete=false
   → 프론트: 답변 표시 + 서식 빈칸 채움

3. 사용자 입력 "to my mother Mary"
   → POST /api/chat (form_state 다시 포함해서!)
   ← ... complete=false ... (계속)

4. 필수 다 차면
   ← form_complete=true
   → 프론트: "PDF 보기" 버튼 활성화

5. 버튼 클릭 → POST /api/render → PDF 받아서 표시
```

## 6. 요약 — 프론트가 신경쓸 것

- ✅ 버튼 → `scenario` 값 정해서 보내기
- ✅ `form_state` 들고 다니며 **매 요청에 다시 보내기**
- ✅ `bot_reply` → 채팅창, `updated_form_state` → 서식 빈칸
- ✅ `form_complete: true` → PDF 버튼 켜기
- ✅ 질문 출처(사람/페르소나) 무관 — 그냥 `user_message`로 보내면 됨

> 별첨: `FIELD_SPEC.md` (각 서식의 전체 필드명 목록)


## ⭐ 추가: 언어 선택 (중요)

프론트에서 시작 전 언어를 선택받으면, **그 값을 매 요청에 `language`로 넣어주세요.**
이러면 사용자가 숫자만 입력해도 챗봇이 선택한 언어로 계속 답합니다.

```json
{
  "scenario": "account_opening",
  "user_message": "20040722",
  "current_form_state": {...},
  "language": "Chinese"
}
```

- `language` 값 예시: "Korean", "English", "Chinese", "Japanese", "Vietnamese", "Indonesian"
- 비워두면("") 백엔드가 발화에서 자동 감지 (단, 숫자만 입력 시 부정확할 수 있음)
- **언어 선택 UI가 있으면 꼭 이 필드로 전달 권장**
